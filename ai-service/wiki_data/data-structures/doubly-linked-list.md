# 双向链表

## 一、定义
双向链表（Doubly Linked List）是一种线性数据结构，其中每个节点包含三个字段：存储数据的`data`域、指向前驱节点的`prev`指针和指向后继节点的`next`指针。与单向链表不同，双向链表允许从任意节点出发，沿两个方向遍历整个序列。

数学上，双向链表可表示为三元组序列：  
`L = ( (d₁, p₁, n₁), (d₂, p₂, n₂), ..., (dₖ, pₖ, nₖ) )`  
其中`dᵢ`为数据元素，`pᵢ`指向`dᵢ₋₁`（前驱），`nᵢ`指向`dᵢ₊₁`（后继），且`p₁ = NULL`，`nₖ = NULL`。

伪代码定义节点结构：
```
Node {
    data: ElementType
    prev: Pointer to Node
    next: Pointer to Node
}
```

## 二、核心原理
1. **双向链接机制**：每个节点同时持有前驱和后继的引用，使得在O(1)时间内可访问任一方向的相邻节点。插入和删除操作只需修改相邻节点的指针，无需遍历整个链表。

2. **对称性**：双向链表的操作具有对称性——在节点前插入与在节点后插入的算法结构完全对称，仅需交换`prev`和`next`的角色。这一性质简化了边界条件的处理。

3. **头尾哨兵节点**：经典实现中常引入哑节点（dummy node）作为头哨兵和尾哨兵，消除对空链表的特殊处理。头哨兵的`next`指向第一个实际节点，尾哨兵的`prev`指向最后一个实际节点。

4. **双向遍历能力**：支持从头部到尾部（正向）和从尾部到头部（反向）的线性扫描，时间复杂度均为O(n)。这一特性在需要频繁访问前驱节点的场景（如LRU缓存淘汰算法）中至关重要。

5. **空间换时间**：相比单向链表，每个节点额外存储一个指针，空间开销增加约50%（在64位系统中每个节点多8字节），但换来了删除指定节点时无需遍历前驱的O(1)时间复杂度。

## 三、过程/示例
考虑一个初始为空的双向链表，依次执行以下操作：在尾部插入10、在尾部插入20、在头部插入5、删除节点20。

**步骤1：初始化**  
创建头哨兵H和尾哨兵T，H.next = T，T.prev = H，H.prev = T.next = NULL。

**步骤2：尾部插入10**  
创建节点A(10)。  
A.prev = T.prev (= H)，A.next = T  
H.next = A，T.prev = A  
链表状态：H ↔ A(10) ↔ T

**步骤3：尾部插入20**  
创建节点B(20)。  
B.prev = T.prev (= A)，B.next = T  
A.next = B，T.prev = B  
链表状态：H ↔ A(10) ↔ B(20) ↔ T

**步骤4：头部插入5**  
创建节点C(5)。  
C.prev = H，C.next = H.next (= A)  
H.next = C，A.prev = C  
链表状态：H ↔ C(5) ↔ A(10) ↔ B(20) ↔ T

**步骤5：删除节点20**  
定位到节点B(20)。  
B.prev.next = B.next (= T)  
B.next.prev = B.prev (= A)  
释放节点B  
链表状态：H ↔ C(5) ↔ A(10) ↔ T

## 四、复杂度/性质分析
**时间复杂度**：
| 操作 | 最好情况 | 最坏情况 | 平均情况 |
|------|---------|---------|---------|
| 访问第i个元素 | O(1) | O(n) | O(n) |
| 在已知节点后插入 | O(1) | O(1) | O(1) |
| 删除已知节点 | O(1) | O(1) | O(1) |
| 按值查找 | O(1) | O(n) | O(n) |

**空间复杂度**：O(n)，每个节点存储3个字段（data + 2指针），常数因子约为单向链表的1.5倍。

**关键性质**：
- **非随机访问**：不支持O(1)下标访问，需从头或尾遍历。
- **缓存不友好**：节点在内存中非连续存储，CPU缓存命中率低于数组。
- **无稳定性概念**：链表操作不涉及相等元素的相对顺序保持问题（排序算法中需注意）。
- **无收敛性**：作为离散数据结构，不涉及迭代收敛过程。

## 五、常见误区
1. **混淆“删除指定节点”与“删除指定值的节点”**：双向链表删除已知节点（持有指针）是O(1)，但删除第一个值为x的节点仍需O(n)查找。初学者常误认为所有删除操作都是O(1)。

2. **指针更新顺序错误**：插入/删除时若先修改某指针导致原引用丢失，会造成内存泄漏或野指针。正确做法是：先建立新节点的前后链接，再断开旧链接。

3. **忽略哨兵节点的边界处理**：不使用哨兵时，在空链表或首尾节点操作需特殊判断。许多实现错误源于忘记更新头指针或尾指针。

4. **与循环链表的混淆**：双向链表的两端指向NULL，而双向循环链表的头尾相连（head.prev = tail, tail.next = head）。前者遍历终止条件为`p == NULL`，后者为`p == head`。

5. **误认为双向链表比单向链表“更好”**：双向链表在删除前驱节点和反向遍历上有优势，但空间开销更大、插入操作指针修改更多。在仅需单向遍历的场景（如队列实现）中，单向链表更优。

# 字典树

## 一、定义
字典树（Trie，又称前缀树或数字查找树）是一种用于高效存储和检索字符串集合的树形数据结构。其核心思想是利用字符串的公共前缀来减少查询时间，避免对每个字符串进行逐字符比较。在字典树中，每个节点代表一个字符（或一个字符的编码），从根节点到某一节点的路径上的字符序列构成一个字符串的前缀。通常，每个节点包含一个布尔标志位，用于指示该节点是否对应一个完整的字符串（即该路径是否构成一个有效词条）。

形式上，设字符集为Σ，字典树T是一个有根树，满足：
- 根节点不包含字符。
- 除根节点外，每个节点包含一个字符c ∈ Σ。
- 从根节点到任意节点p的路径上的字符序列构成字符串s(p)。
- 若节点p的标志位为真，则s(p)属于字典集合。

字典树的插入与查询操作均基于逐字符遍历路径，其核心数学描述为：对于字符串s = c₁c₂...cₖ，从根节点出发，依次查找子节点c₁、c₂、...、cₖ，若路径存在则继续，否则创建新节点。查询时，若路径存在且末节点标志位为真，则字符串存在。

## 二、核心原理
1. **前缀共享机制**：字典树通过合并公共前缀来节省存储空间。例如，字符串“cat”和“car”共享前缀“ca”，在树中仅需一条路径表示“ca”，然后分叉为“t”和“r”。这种机制使得字典树在处理大量具有公共前缀的字符串时，空间效率优于哈希表或平衡树。

2. **逐字符匹配**：查询或插入操作均从根节点开始，按字符串的字符顺序逐层向下移动。每个节点仅存储一个字符，且子节点通常通过数组或哈希表实现快速查找（时间复杂度O(1)）。这种设计使得操作时间仅与字符串长度相关，与字典大小无关。

3. **节点标志位**：每个节点可携带一个布尔标志（isEnd），用于标记从根到该节点的路径是否构成一个完整词条。这一设计允许字典树同时存储前缀和完整字符串，例如“app”和“apple”可共存，其中“app”路径上的节点标志位为真，而“apple”路径上的末节点标志位也为真。

4. **字符集依赖**：字典树的性能与字符集大小|Σ|密切相关。若|Σ|较小（如26个英文字母），每个节点可使用固定大小数组存储子节点指针，空间开销可控；若|Σ|较大（如Unicode），则需使用哈希表或动态数组，可能增加时间开销。

5. **有序性**：字典树天然支持字典序排序。通过深度优先遍历（按字符顺序访问子节点），可输出所有字符串的有序序列。这一性质在自动补全、拼写检查等场景中极为有用。

## 三、过程/示例
考虑字符串集合S = {"cat", "car", "dog"}，字符集Σ = {a, c, d, g, o, r, t}。构建字典树的过程如下：

1. **插入"cat"**：从根节点出发，查找子节点'c'，不存在则创建。进入节点'c'，继续查找子节点'a'，不存在则创建。进入节点'a'，查找子节点't'，不存在则创建。进入节点't'，设置其isEnd = true。此时路径为root → c → a → t。

2. **插入"car"**：从根节点出发，查找子节点'c'，存在（步骤1创建）。进入节点'c'，查找子节点'a'，存在。进入节点'a'，查找子节点'r'，不存在则创建。进入节点'r'，设置isEnd = true。此时路径为root → c → a → r，与"cat"共享前缀"ca"。

3. **插入"dog"**：从根节点出发，查找子节点'd'，不存在则创建。进入节点'd'，查找子节点'o'，不存在则创建。进入节点'o'，查找子节点'g'，不存在则创建。进入节点'g'，设置isEnd = true。路径为root → d → o → g。

最终字典树结构如下（括号内表示isEnd状态）：
- root
  - c
    - a
      - t (true)
      - r (true)
  - d
    - o
      - g (true)

查询"ca"：从根节点出发，依次找到c和a，但节点a的isEnd为false，因此"ca"不是有效词条。查询"car"：路径存在且节点r的isEnd为true，返回存在。查询"dog"：路径存在且节点g的isEnd为true，返回存在。查询"cat"：路径存在且节点t的isEnd为true，返回存在。

## 四、复杂度/性质分析
- **时间复杂度**：
  - 插入与查询：O(|s|)，其中|s|为字符串长度。该时间与字典中字符串总数N无关，仅取决于字符串本身长度。最好、最坏、平均情况均为O(|s|)。
  - 删除操作：需先查询字符串是否存在，然后递归删除节点，时间复杂度同样为O(|s|)。
  - 前缀查询（如查找所有以"ca"开头的字符串）：需先定位前缀路径（O(|前缀|)），然后遍历子树，输出所有isEnd为真的节点。遍历子树的时间与子树大小成正比，最坏情况下为O(N·|s|)。

- **空间复杂度**：
  - 最坏情况：所有字符串无公共前缀，则每个字符对应一个节点，节点总数等于所有字符串字符数之和，即O(∑|sᵢ|)。每个节点需存储子节点指针（|Σ|个指针或哈希表），因此空间复杂度为O(∑|sᵢ|·|Σ|)（数组实现）或O(∑|sᵢ|)（哈希表实现，但哈希表本身有额外开销）。
  - 最好情况：所有字符串完全相同，则仅需一条路径，节点数为|s|+1（含根节点），空间复杂度为O(|s|·|Σ|)。
  - 平均情况：取决于字符串的公共前缀长度，通常介于两者之间。

- **稳定性**：字典树本身不涉及排序操作，但若按字符顺序遍历子节点，则输出结果稳定（与插入顺序无关）。

- **收敛性**：字典树是一种确定性的数据结构，其构建过程不涉及迭代或近似，因此不存在收敛性问题。

## 五、常见误区
1. **混淆字典树与二叉搜索树**：二叉搜索树（BST）基于键值比较，每个节点存储完整键，且左右子树分别包含小于/大于当前键的节点。字典树则基于字符分解，每个节点仅存储一个字符，且路径隐含键值。BST的查询时间为O(log N)，而字典树为O(|s|)，在字符串长度较短且N极大时，字典树更优。

2. **忽略字符集大小的影响**：初学者常假设子节点查找为O(1)，但若字符集较大（如Unicode），使用固定数组会导致巨大空间浪费，而使用哈希表则可能引入哈希冲突，实际时间复杂度可能退化。需根据字符集特性选择实现方式。

3. **误认为字典树只能存储字符串**：字典树可推广至任意序列，如整数（按位分解）、DNA序列（字符集{A, C, G, T}）等。其核心是“前缀共享”，与具体数据类型无关。

4. **忽视节点标志位的作用**：若未正确设置isEnd标志，可能导致前缀被误认为完整词条。例如，插入“apple”后查询“app”，若节点p的isEnd为false，则返回不存在，但若忘记设置标志位，可能错误返回存在。

5. **认为字典树空间效率总是优于哈希表**：当字符串无公共前缀时，字典树节点数等于总字符数，每个节点还需存储指针，空间开销可能远超哈希表（哈希表仅存储完整字符串）。字典树的空间优势仅体现在大量公共前缀场景下。

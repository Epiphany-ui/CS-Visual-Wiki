# 哈希表

## 一、定义

哈希表（Hash Table），又称散列表，是一种通过键（Key）直接访问存储在内存中位置的数据结构。其核心思想是利用一个哈希函数（Hash Function）将键映射为数组的索引，从而在理想情况下实现常数时间复杂度的插入、删除和查找操作。

**核心数学定义**：  
设 \( U \) 为所有可能键的宇宙，\( K \subseteq U \) 为实际存储的键集合，哈希表是一个数组 \( T[0..m-1] \)，配合哈希函数 \( h: U \rightarrow \{0, 1, ..., m-1\} \)。对于键 \( k \in K \)，其存储位置为 \( T[h(k)] \)。

**伪代码（查找操作）**：
```
HashSearch(T, k):
    index = h(k)
    if T[index] == k:
        return index
    else:
        // 处理冲突（如链地址法）
        return SearchInChain(T[index], k)
```

## 二、核心原理

1. **哈希函数映射**：哈希函数将任意大小的输入键压缩到固定范围的整数索引。理想哈希函数应满足：计算简单、分布均匀、冲突概率低。常见方法包括除法散列法（\( h(k) = k \mod m \)）和乘法散列法。

2. **冲突解决机制**：由于 \( |U| \gg m \)，不同键可能映射到同一位置，即冲突。主要解决策略有两种：
   - **链地址法（Chaining）**：每个槽位维护一个链表，冲突元素存入链表。
   - **开放地址法（Open Addressing）**：冲突时探测下一个空槽，如线性探测（\( h(k,i) = (h(k) + i) \mod m \)）、二次探测、双重散列。

3. **负载因子**：定义为 \( \alpha = n/m \)，其中 \( n \) 为已存储元素数，\( m \) 为槽位数。负载因子直接影响性能，通常当 \( \alpha \) 超过阈值（如0.75）时进行**动态扩容**（Rehashing）。

4. **动态扩容**：当负载因子过高时，分配更大的新数组（通常为原大小的2倍），重新计算所有键的哈希值并插入新表。这是保证均摊性能的关键机制。

## 三、过程/示例

假设我们有一个大小为 \( m = 7 \) 的哈希表，使用除法散列函数 \( h(k) = k \mod 7 \)，冲突解决采用链地址法。依次插入键：50, 21, 58, 17, 15。

**步骤1**：插入50  
\( h(50) = 50 \mod 7 = 1 \)，槽位1为空，直接插入。

**步骤2**：插入21  
\( h(21) = 21 \mod 7 = 0 \)，槽位0为空，插入。

**步骤3**：插入58  
\( h(58) = 58 \mod 7 = 2 \)，槽位2为空，插入。

**步骤4**：插入17  
\( h(17) = 17 \mod 7 = 3 \)，槽位3为空，插入。

**步骤5**：插入15  
\( h(15) = 15 \mod 7 = 1 \)，槽位1已有50，发生冲突。采用链地址法，将15链接到50之后。

最终哈希表状态：
```
索引0: [21]
索引1: [50] -> [15]
索引2: [58]
索引3: [17]
索引4: 空
索引5: 空
索引6: 空
```

**查找过程**：查找15，计算 \( h(15)=1 \)，遍历索引1的链表，找到15，返回成功。查找22，计算 \( h(22)=1 \)，遍历链表未找到，返回失败。

## 四、复杂度/性质分析

**时间复杂度**（假设哈希函数均匀分布）：
- **平均情况**：插入、删除、查找均为 \( O(1) \)（链地址法下 \( O(1+\alpha) \)，当 \( \alpha \) 为常数时即 \( O(1) \)）。
- **最坏情况**：所有键映射到同一槽位，退化为链表，操作复杂度 \( O(n) \)。
- **均摊分析**：考虑动态扩容，插入操作的均摊时间复杂度仍为 \( O(1) \)。

**空间复杂度**：\( O(n + m) \)，其中 \( n \) 为元素数，\( m \) 为槽位数。链地址法需要额外链表指针空间。

**关键性质**：
- **无序性**：哈希表不维护键的顺序，遍历顺序与插入顺序无关。
- **非稳定性**：扩容后所有元素的存储位置可能改变，迭代器失效。
- **无收敛性问题**：哈希表操作是确定性的，不存在迭代收敛问题。

## 五、常见误区

1. **混淆哈希表与平衡二叉搜索树**：哈希表提供平均 \( O(1) \) 的查找，但无序；平衡树（如红黑树）提供 \( O(\log n) \) 的有序操作。需要范围查询时应选择树结构。

2. **忽视哈希函数质量**：认为任何哈希函数都行。实际上，不良哈希函数（如取模时 \( m \) 为2的幂）会导致大量冲突，性能退化至 \( O(n) \)。建议 \( m \) 取素数，或使用成熟的哈希算法。

3. **误以为哈希表总是 \( O(1) \)**：最坏情况下（如所有键冲突）复杂度为 \( O(n) \)。实际应用中需通过良好哈希函数和合理负载因子控制风险。

4. **混淆键与值的概念**：哈希表存储的是键值对（Key-Value Pair），查找时通过键定位，返回对应的值。初学者常误以为哈希表只存储键本身。

5. **忽略扩容代价**：认为插入永远是 \( O(1) \)。实际上单次扩容操作是 \( O(n) \)，但均摊后为 \( O(1) \)。在实时系统中需谨慎处理扩容带来的延迟峰值。
